Privacy & Security Privacy & Security

This page provides a comprehensive analysis of privacy and security considerations for this mobile inventory management application, aligned with PROG2005 Programming Mobile Systems and the Australian Privacy Act 1988.

Core Security Requirements

Input Validation & Data Integrity

  • Strict client-side input validation enforces type safety: item names and supplier names accept only letters, spaces, and hyphens; quantity and price accept only positive integers
  • TypeScript interfaces and enums guarantee that all data conforms to the predefined schema (InventoryItem, Category, StockStatus), preventing type-related vulnerabilities at compile time
  • Server-side validation on the REST API provides a secondary defence layer, rejecting malformed or invalid payloads
  • Real-time field validation provides immediate user feedback, reducing the chance of submitting erroneous data

Secure Data Transmission

  • All communication with the server uses HTTPS (TLS encryption) to protect data in transit against eavesdropping, tampering, and man-in-the-middle attacks
  • JSON-formatted API requests follow RESTful conventions with proper Content-Type headers
  • No sensitive authentication tokens or API keys are stored on the device, minimising the attack surface if the device is compromised
  • URL encoding (encodeURIComponent) is applied to item names in API requests to prevent injection via URL manipulation

User Action Confirmation

  • Delete operations require explicit user confirmation via Ionic AlertController dialogs, preventing accidental data loss
  • All data modification actions (add, update, delete) provide immediate visual feedback through Toast notifications
  • The server enforces business rules such as forbidding deletion of the "Laptop" item, ensuring at least one record persists for verification
 Privacy Considerations & Australian Privacy Act

Australian Privacy Principles (APPs) Compliance

  • APP 3 (Collection of personal information): This application does not collect personal information. Only business inventory data (item names, quantities, prices) is transmitted to the server
  • APP 6 (Use or disclosure of personal information): Since no personal information is collected, there is no risk of misuse or unauthorised disclosure
  • APP 8 (Cross-border disclosure): All data remains on an Australian university-managed server (prog2005.it.scu.edu.au), with no cross-border data transfer
  • APP 11 (Security of personal information): While the data handled is not personal information, the same security standards (HTTPS, input validation, access controls) are applied to all data

Notifiable Data Breaches (NDB) Scheme

  • Although this educational application does not handle personal information, the architecture supports compliance with the NDB scheme should personal data be introduced in future versions
  • HTTPS encryption and server-side access controls would form the basis of "reasonable steps" to protect data as required by the NDB scheme

Mobile-Specific Privacy Risks Mitigation

  • The app does not request unnecessary device permissions (camera, location, contacts, storage), following the principle of data minimisation
  • No background data processing — the app only transmits data when the user actively performs an action
  • Compatible with mobile OS privacy features including app permissions management and data clearing (iOS Settings, Android App Info)
  • Ionic framework ensures the app runs in a secure WebView container with the standard browser security model (same-origin policy, CSP)
  • No analytics or tracking libraries (e.g., Google Analytics, Firebase) are integrated, eliminating third-party data collection

Data Storage on Remote Server

  • Inventory data is stored on a university-managed server (prog2005.it.scu.edu.au) governed by institutional IT security policies
  • Users should be aware that inventory data is stored remotely and could be accessed by server administrators or other students on the shared server
  • It is strongly recommended not to enter real sensitive business data in this educational application
 Ionic & Capacitor Mobile Architecture Security

Hybrid Mobile Architecture Security Model

  • WebView sandboxing: The Ionic app runs within a sandboxed WebView, inheriting browser security features such as same-origin policy, Content Security Policy (CSP), and cookie isolation
  • Capacitor secure bridge: Capacitor provides a secure bridge between the web layer and native platform APIs, ensuring that native functionality (haptics, keyboard, status bar) is accessed through well-defined, auditable interfaces
  • Code signing: When deployed to app stores, the application package is code-signed, preventing tampering and assuring users of the app's authenticity
  • No local storage of credentials: The application does not store any login credentials, API keys, or tokens on the device, eliminating the risk of credential theft from local storage

Native Device Features (Capacitor Plugins)

  • Haptics: The app uses Capacitor Haptics for tactile feedback on user actions, demonstrating mobile-native integration without compromising security
  • Keyboard management: Capacitor Keyboard plugin handles native keyboard interactions, ensuring input fields behave consistently across platforms
  • Status bar: Capacitor StatusBar plugin provides native status bar configuration for a seamless mobile experience
  • All native plugins are accessed through Capacitor's permission model, which follows platform-specific security conventions
 Ongoing Security Best Practices
  • Regular dependency updates: Ionic 8, Angular 20, and Capacitor 8 dependencies are kept up to date to patch known CVEs (Common Vulnerabilities and Exposures)
  • Secure coding practices: Angular's built-in template sanitisation prevents XSS (Cross-Site Scripting) attacks; TypeScript's static type checking prevents type confusion vulnerabilities
  • API error handling: Proper error handling on both client and server prevents sensitive server information (stack traces, database schemas) from being exposed to the client
  • Cross-Origin Resource Sharing (CORS): The server API is configured to only accept requests from authorised origins, preventing unauthorised cross-origin access
  • User education: This privacy page and the contextual help widget on every page provide transparent guidance on secure app usage
  • Principle of least privilege: The app only requests the minimum device capabilities needed, and all API operations are scoped to the specific inventory data model
 Security FAQ What data does this app store on my device?
This application does not store any data locally on your device. All inventory data is fetched from and sent to the remote server (prog2005.it.scu.edu.au) in real time. No cookies, local storage, or SQLite databases are used.
Is my data encrypted?
All data in transit between the app and the server is encrypted using HTTPS (TLS). However, data at rest on the server is managed by the server's security policies. You should not enter sensitive business data as the server is shared among students.
What happens if I lose my device?
Since no data is stored locally on the device, there is no risk of data exposure from device loss. Your inventory data remains safely on the server and can be accessed from any device running this app.
How does the app prevent unauthorised modifications?
The app uses strict input validation (names accept only letters, numbers only accept integers), confirmation dialogs for destructive actions (delete), and server-side business rules (e.g., "Laptop" cannot be deleted). These layers of protection prevent both accidental and malicious data modifications.
